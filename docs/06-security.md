# Security Checklist

Tracked posture. Each item: current state, target state, and the phase that lands it. ☑ = done; ☐ = not yet; ☒ = legacy state still present.

## Secrets management

- ☑ `.env` removed from git tracking (Round 1)
- ☑ `server/.env.example` created and authoritative (Round 1)
- ☑ Root `.gitignore` excludes `.env`, `.env.*`, `*.pem`, `*.key`, `dump.rdb`, `uploads/` (Round 1)
- ☒ **MongoDB credentials and Google OAuth Client Secret were previously committed.** The owner must rotate both immediately. If the repo is or will be public, also rewrite history with `git filter-repo` or `BFG` to scrub the old secrets from past commits.
- ☑ Google OAuth credentials read from `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` (Round 2 — `src/modules/auth/google.ts`)
- ☑ Session secret validated at boot (≥ 32 chars); Zod rejects insecure defaults (Round 2)
- ☑ Production secrets flow through the host's env layer — documented in `docs/08-deployment.md` (Round 8). `.env.example` is the template; real values go in Railway/Render/Fly variables or Docker Compose `env_file`.

## Authentication

- ☑ Passwords hashed with bcrypt, cost **12** (Round 2 — `auth.service.ts`)
- ☑ Session IDs generated by `express-session` (128-bit crypto random) (Round 2)
- ☑ Sessions persisted to Mongo via `connect-mongo` (Round 2 — `config/session.ts`)
- ☑ Session regenerate on login + on password change to prevent fixation (Round 2)
- ☑ Google OAuth state parameter verified (Passport default, confirmed) (Round 2)
- ☑ Password change invalidates all other sessions for that user (Round 2 — `destroyOtherSessions`)
- ☐ Email verification required for sensitive actions (Round 7 — out of scope earlier)
- ☑ Login rate limit: 5/5min per IP+username (Round 2)
- ☑ Register rate limit: 3/hour per IP (Round 2)
- ☑ Generic login error message — no enumeration (Round 2)

## Authorization

- ☑ All state-changing routes behind `requireUser` middleware (Round 2)
- ☑ Ownership checks in the service layer, not the controller (Round 2 establishes pattern)
- ☑ Visibility rules (public/followers/private) enforced in service layer (Round 3 — `posts.service.assertVisibility`)

## Input validation

- ☑ **Zod schemas for every request body, params, query** (Round 2 — `middlewares/validate.ts`)
- ☐ `express-mongo-sanitize` mounted globally to strip `$` and `.` from user input (Round 3 — not strictly needed yet; Zod catches most, but defense-in-depth)
- ☑ File upload: max 5MB, MIME `image/*` check via multer fileFilter (Round 2)
- ☑ Image uploads processed by Cloudinary (buffer streamed, no disk hop) (Round 2 — `config/cloudinary.ts`)
- ☑ Markdown render sanitized with `DOMPurify` (client) — Round 7. Strict `ALLOWED_TAGS` allowlist, no event handlers, `javascript:` URLs stripped, `rel="noopener noreferrer nofollow"` auto-set on external anchors.

## Transport & headers

- ☑ `helmet()` mounted globally (Round 2)
- ☑ Strict CSP (Round 8) — explicit allowlist, no unsafe-inline scripts; `'unsafe-eval'` only in dev for Vite HMR
- ☑ CORS configured via `CORS_ORIGINS` env allowlist; credentials: true (Round 2)
- ☑ `trust proxy` set (Round 2)

## Cookies

- ☑ `sid` cookie: HttpOnly, SameSite=Lax, Secure (env-driven), signed (Round 2)
- ☑ Clear cookie on logout (Round 2)

## Rate limiting

- ☑ Global: 100 req/min per IP (Round 2)
- ☑ Auth endpoints: stricter limits wired (Round 2)
- ☑ Write endpoints — per-user rate limit (Round 7): posts 30/min, comments 60/min, messages 60/min via `middlewares/rateLimit.ts` helpers keyed by `req.user.id`

## Logging & monitoring

- ☑ `pino-http` structured request log; redacts `authorization`, `cookie`, `set-cookie`, `password*`, `email` (Round 2)
- ☑ Error logs include `requestId`; `requestId` also in the error envelope (Round 2)
- ☑ No secrets, no full request bodies in logs (Round 2)
- ☑ Sentry scaffolding on both ends (Round 8) — env-gated; wire-up described in `monitoring.ts` in each package. Installing `@sentry/node` + `@sentry/react` and setting DSN env is the turn-key.
- ☐ Login success/failure audit events (Round 7)

## Data protection

- ☑ Soft-delete fields on User (`status`, `deletedAt`) (Round 2)
- ☑ Soft-delete on Post and Comment (`status`, `deletedAt`); read path filters `status: 'active'` (Round 3)
- ☑ PII-minimal stance maintained: no IP, UA, fingerprints stored (Round 2)
- ☑ Backups runbook (Round 8) — `docs/08-deployment.md` "Backups" section: Atlas snapshots for cloud; nightly `mongodump` cron for self-host; retention 30 days; restore command documented

## Dependencies

- ☐ `npm audit` clean at each phase close (ongoing — Round 2 has non-critical warnings from multer/pino peer deps)
- ☑ Dependabot enabled (Round 8) — `.github/dependabot.yml` with grouped updates for React / TanStack / Radix / types
- ☑ Pin `engines.node >=20.10` in `package.json` (Round 2)
- ☑ Legacy `request` package removed (Round 2)

## Realtime

- ☑ Socket.io handshake reuses session cookie via `io.engine.use(sessionMiddleware)` — no separate token (Round 6)
- ☑ Unauthenticated sockets rejected at connection (Round 6 — `server/src/realtime/io.ts`)
- ☑ `conversation:join` checks membership server-side before joining the room (Round 6)
- ☑ Zod validation on inbound socket events — `conversation:join/leave` payloads (Round 7; malformed → silent drop / ack:false)

## Uploads

- ☑ Multer with `memoryStorage`, buffer straight to Cloudinary, no disk hop (Round 2)
- ☑ 5 MB per file cap at multer level (Round 2)
- ☑ Max 4 images per post (Round 3 — enforced at multer `files: 4` + `.array('images', 4)`)

## Rotation & incident response

- ☑ Secret-rotation runbook (Round 8) — in `docs/08-deployment.md` "Secret rotation" section: SESSION_SECRET, Mongo password, Google OAuth, Cloudinary, + force-session-wipe command
- ☐ If a secret leaks: rotate immediately, invalidate all sessions (`db.sessions.deleteMany({})`), force re-login. Document here post-v1.

## Scrubbing history (the one-time thing)

The Mongo password and Google OAuth secret are in git history. The owner must:

1. Rotate both secrets NOW.
2. If the repo will be public: `git filter-repo --path server/.env --invert-paths` and force-push (or use BFG). Coordinate with anyone who has cloned.
3. Confirm Atlas access logs don't show usage from unknown IPs during the exposure window.

Until this is done, **do not make the repo public.**
